
# Offline-First Fail-Safes, LocalStorage Synchronization, and UX Network Polishing

This document outlines the strict production-grade rules and engineering patterns to implement comprehensive client-side resiliency across the entire dashboard ecosystem. The objective is clear: when the network link drops entirely, the application must intercept failures gracefully, fall back to historical device cache, and alert the clinician via non-blocking notifications.

## 2. Technical Component Specifications

### A. The Global Network Status Monitor Hook (`hooks/use-network-status.ts`)

* **State Detection:** Utilize native browser APIs (`navigator.onLine`) bound to global window event listeners (`online`, `offline`) to track the exact connectivity envelope of the device in real-time.
* **Global Broadcast:** Expose an immutable boolean tracking flag (`isOnline`) so any form or diagnostic card can instantly modify its behavior, submission routes, or visual state tags dynamically.

### B. Request Interception & LocalStorage Storage Fallbacks

Every functional core module must implement a dual-path execution lifecycle:

1. **The Diagnostic Matrix:** When executing a risk assessment offline, fall back to historical evaluations cached under `localStorage.getItem('cached_risk_assessments')`.
2. **The Immunization Ledger:** When registering a vaccination offline, append the structural payload array straight to a localized queue (`offline_vaccination_queue`). The moment connectivity re-establishes, drain the queue safely via batch API synchronizations.
3. **The Guidance Checklist:** Store checked-off state variables inside local device memory immediately so an unexpected page refresh deep in the field doesn't clear the active checklist history.

### C. Low-Resource UX Polishing (Skeletons & Toasts)

* **High-Latency Skeletons:** To prevent layout shifts on unstable 2G/3G connections, replace simple spin loaders with precise structural component wireframe blocks (`<Skeleton>` from Shadcn/UI) matching the exact dimensions of incoming data tables.
* **Notification Layer:** Handle every failure or fallback intercept using highly accessible popup notifications (`toast()` out of the Shadcn Toast engine):
* **Offline Active:** Muted slate info toast reading *"Network disconnected. Switched to offline cache mode."*
* **Sync Complete:** Vibrant green success toast reading *"Connectivity restored. Successfully synced field records with database server."*
