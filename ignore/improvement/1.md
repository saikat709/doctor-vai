## TASK: Add Multi-Language Support using `next-intl`

### Git Setup
First, create and switch to a new branch:
```
git checkout -b feature/multi-language-support
```

---

### 1. Install Dependencies

```bash
npm install next-intl
```

---

### 2. Supported Locales

Support the following locales:
- `en` — English (default fallback)
- `bn` — Bengali / Bangla
- `hi` — Hindi
- `ur` — Urdu

---

### 3. Locale Detection & Default Selection

- Auto-detect locale from the browser's `Accept-Language` header on first visit using `next-intl` middleware.
- If the detected locale is not in the supported list, fall back to `en`.
- Store the user's manually selected locale in `localStorage` as `"preferred_locale"` and also sync it to the user's DB record (see Prisma schema update below).
- Once the user has a stored preference, always use that over browser detection.

---

---

### 5. next-intl Configuration

**`i18n.ts` (or `i18n/request.ts`):**
Configure `next-intl` with the `getRequestConfig` pattern for App Router.

**`middleware.ts`:**
Add `next-intl` middleware that:
- Matches all routes except `_next`, `api`, and static files
- Reads `preferred_locale` cookie first, then falls back to `Accept-Language` header
- Redirects to the appropriate locale-prefixed route

Use the `[locale]` dynamic segment pattern under `app/[locale]/` for all pages.

---

### 6. App Router Restructure

Wrap the entire `app/` directory under `app/[locale]/` so that:
- `app/[locale]/layout.tsx` — root layout with `NextIntlClientProvider`
- `app/[locale]/page.tsx` — landing page
- `app/[locale]/dashboard/...` — all dashboard routes
- `app/[locale]/(auth)/...` — auth routes

Maintain all existing route functionality. Update all `Link` hrefs to be locale-aware using `next-intl`'s `useRouter` and `Link` (from `next-intl/navigation`).

---

### 7. Language Switcher in Header

Add a language switcher dropdown to the **top header bar** of the dashboard layout:

- Show the current language's flag emoji + short label (e.g., 🇧🇩 বাংলা, 🇬🇧 English, 🇮🇳 हिंदी, 🇵🇰 اردو)
- Use a Shadcn/UI `DropdownMenu` component
- On selection: update `localStorage`, set a `preferred_locale` cookie (for middleware), call a `PATCH /api/user/preferences` endpoint to persist to DB, and use `next-intl`'s `useRouter().replace()` to switch locale without full reload

---

### 8. Chatbot / Assistant Language Support

In the AI assistant (`/dashboard/assistant` or wherever the RAG chatbot lives):

- Read the user's `chatbotLanguage` preference from DB (see schema update)
- Inject the language instruction into the system prompt: `"Always respond in [language]. If the user writes in another language, still respond in [language]."`
- Language names for the prompt: English → `"English"`, Bangla → `"Bengali (Bangla)"`, Hindi → `"Hindi"`, Urdu → `"Urdu"`
- Also apply this to any other AI calls (risk detection, medicine interaction) — add a language instruction to each system prompt based on user preference

---

### 9. Prisma Schema Update

Add the following fields to the `User` model (or a related `UserSettings` model if one exists):

```prisma
preferredLocale   String  @default("en")
chatbotLanguage   String  @default("en")
```

Generate and run the migration:
```bash
npx prisma migrate dev --name add_language_preferences
```

---

### 10. Settings Page — Language Section

In the existing Settings page (`/dashboard/settings`), add a **"Language & Region"** section before the existing file upload section:

**UI Language:**
- A `Select` component (Shadcn/UI) showing the 4 supported locales
- Label: "Interface Language"
- On change: same locale switching logic as the header dropdown

**Chatbot Response Language:**
- A separate `Select` component
- Label: "AI Response Language"
- Options: English, বাংলা, हिंदी, اردو
- On save: `PATCH /api/user/preferences` with `{ chatbotLanguage: value }`
- Show a toast confirmation on save

## Key Constraints

- Do NOT break existing functionality — all current routes, auth flows, and AI calls must continue working
- Use `next-intl` patterns only (no `i18next`, no `react-i18next`)
- All translation keys must cover the full UI — no hardcoded English strings should remain in any component
- The `[locale]` prefix must be transparent to the user (middleware handles it, no manual locale in every `Link`)
- RTL support is NOT required for this phase (Urdu RTL can be Phase 3)
